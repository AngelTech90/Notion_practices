Initial config:

When we open our terminal is important know that when we create git repository it will have our iD that is compound of our alias and Gmail.

💥In git, we have 3 levels of herarchy, local, global and system.

💥A simple dir have a local reach for example.

💥All our OS user inside have a global reach.

💥All inside our OS have a system reach.

💥With "git config" we can set the reach that we will use to work:

git config --local

git config --global

git config --local

💥We can config our global user name in git using:

git config --global user.name "git user name"

💥And for set our mail:

git config --global user.email "git mail"

💥When we work with git we usually work with a code editor too, we have to config were code editor we'll use:

For put like editor vs code:

git config --global core.user "code --wait"

The "--wait" is for load our editor once we stop to use it.

💥If we want to activate our colors in our IU we write:

git config --global color.ui true

💥Basically in windows we have 2 special and hidden chars for give a line jump and a reset with some concept called carriage return linefield.


💥(crlf) this makes a line jump but returning to the start of where we start to write.

💥This concept is more the concat of the linefield that is for only go down the currently writing line, and the carriage return, that take
us again to the start of the line.

💥This is a special feature of windows operative system , beacause in linux or mac works diferent.

💥"\r" in windows is the carriage return.

💥"\n" in windows is the linefield.

💥This is important because when we will work with linux or mac repos we have to get the carriage return and our linefield in our git for
don't have troubles working with that.

💥We write if we are using windows this:

git config --global core.autocrlf true

💥We write this if we are in linux or mac:

git config --global core.autocrlf input

First steps in git

Creating and understanding our first repo:

Some important details:

💥The staging area is for valid where our changes of our project is ready for upload our server.

💥The final repo is the part were is our final project.

How files works in git:

Creating first repo:

💥For create a git repo we enter in directory were we need to use it and write:

git init

Making this we create all the internal structure in our dir for manage our git repository.

💥Git bash is based in Linux, so we move equal in this terminal.

A git branch is like a time line.

💥For put inside a file in the staging area we write:

git add name_file

💥Example:

git add index.js

💥For see how is our git status:

git status

💥For remove a file of the stagging area:

git rm —cached prove.py

💥All files in red when see our git status are out of our work flow in git.

Making our first commit in git:

💥We write for commit a new file in repo:

git commit

💥The commit works uploading all in stagging area to repo

💥For add a message in our commit we can use the wildcard -m

git commit -m “Hi, there’s our first commit”

The git commits have to get a lot of details like:

💥What commited file do.

💥Reasons for commit

💥And general details of file in the project git repo

💥If we want to directly commit our file we use wildcard -a:

git commit -a

💥With this we don’t need to add our files in the staging area.

Managing the commits in git:

💥For restore the files to the work area we use:

git restore

💥For check our modified archive and see if our archive get the changes we use:

git checkout prove.py

💥Git checkout works checking the last state of our project after the last commit and we restore the archive in our work area.

💥For reset the changes in our staging area and our commit area:

git reset —hard

💥For get a shortened vision of git status we write:

git status -s

or

git status —short

💥For see the code inside our committed file we use:

git show prove.py

💥For compare the differences in the staged area and the work area:

git diff —staged

For compare the differences between commits we have to get a id for every commit:

💥If we want to see tall he commits ID:

git log

💥If we want to see all our commits in oneline:

git log —oneline

💥Every commit have a unique hash id, for avoid that it be have to manipulate in our global config the amount of chars that could be showed abbvred in a one line commit id (usually this is important in very big repos, in the most cases 5 chars are suficent).

git config —global core.abbrev 5

💥If we now see the abbvrev ID we can use it for compare the commits:

git diff h8j29 92jdc

💥Here we are comparing 2 differents commits (5 chars every abbvreved id).

💥For only see the name of the files that have any difference we use:

git diff h8j29 92jdc —name-only

💥For see directly the change in the chars in our commits:

git diff —word-diff

Modify and delete a commit

Is not recommended modify the last commit because we are removing the last element for generate a new commit to add to our master branch.

We can use this ONLY with the last commits:

git commit -amend

Use git reset:

we can use —soft wild card for move the head of our branch of our commit area and let a copy in the staging area for be modified and commited:

git reset —soft e4rxt

If we want reset our head an specific amount of commits back we do:

git reset —soft head~3

3 is the amount of commits that we go back.

We can use git reset mixed for remove all commits after the selected new head:

git reset —mixed 34af3

With git reset hard we do this:

1.We delete all commits after we select our new head.

2.We delete all information in our staging area.

3.We replace and overwrite if is necesary the info of work area with the info of the deleted commit.

Git branches:

The concept that make git different of other versions control systems is the branches concepts.

A branch in git is basically a pointer that check the commits time lines in our repository.

A good analogy is see a branch like other copy of our project, were we keep a flow of work that don’t affect the master branch

And we can make a “merge” that in git is fusion a branch with the master branch of our project.

A branch is a very temporal version of the project that only exist for a exact purpose, like solve an error, optimize a code or in general the normal tasks that we can see in our project.

And then when we solve the task that we use to create the branch we make a merge in implement the changes to the project.

And when we finish the task we can erase the branch that we create to solve the problem.

Usually, for name our git brances we use the kebab-case:

first-git-branch

Creating our first branch in git:

We write:

git branch new-branch

and for see the total branches in our project:

git branch

if we want to move in our project branches we use:

git switch new-branch

Usually when we create a git branch we want to move directly to the branch because we want to solve a problem, for don’t have to use 2 commands and go to the new branch we write:

git switch -c new-brach

We can delete a branch writing:

git branch -d new-branch

Is very important that when we delete a branch we been in the master branch

For modify the branch name if we aren’t in the branch that we want to modify we write:

git branch -m new-branch new-name-branch

And if we are inside the branch that we want to modify we write:

git branch -m new-branch-name-two

How we can merge the git branches?:

First we’ll go to the master branch, and then we write:

git merge new-branch-name-two

And now we have all the commits merged in our master branch (This apply to others branches that we want to merge).

In the case that we make a mistake merging the branches or something we have the reset command, and we’ll reset our branch to the last commit that we merge:

git reset —hard 8jnm3
